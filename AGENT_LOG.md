# Agent Behavior & Correction Log

This file tracks policy violations, behavioral corrections, and manual overrides of AI agent actions in this repository. Agents MUST read this log at the beginning of every interaction to learn from past mistakes.

- **Planka SSO bridge used fragile iframe form-fill instead of browser-native fetch**: `planka_redirect` built a bridge page that created a hidden `<iframe src="/login">`, polled for React form fields every 20ms, and simulated input events to click the submit button. This broke with browser security policies (same-origin cross-frame access restrictions), newer versions of Planka changing their React form selectors, and the 75ms blind timeout after `btn.click()` being too fast for the login response. The `httpOnlyToken` cookie was also proxied through the backend HTTP client rather than set natively by Planka in the browser's own request. Fix: replaced the entire iframe approach with a browser-side `fetch('/api/access-tokens?withHttpOnlyToken=true', {credentials: 'include'})` call. Traefik routes `/api/*` (excluding `/api/dashboard`) to Planka, so the browser makes the auth request directly and Planka sets its httpOnly cookie natively in the browser's cookie jar. The backend-obtained token is pre-seeded in localStorage (`accessToken`, `token`) as an immediate fallback. ToS 403 acceptance is also handled in the browser-side fetch path. Rule: NEVER use iframe form-fill to log into a React SPA. Always have the browser make the auth API call directly with `credentials: 'include'` so the server sets cookies natively.

- **LangGraph agent fallthrough silently swallowed conversational non-tool responses (llm.py `chat_with_context`)**: When user said "Job application sent. Remind me today to order Marlos presents", Z responded conversationally ("Okay, I've noted...") with no changes on the operator board. Root causes: (1) The `needs_agent` LangGraph path only fell back to direct chat when the reply contained specific error marker strings — if the local LLM simply responded conversationally without calling any tools (common with llama.cpp models that don't reliably support tool-calling), the hallucinated response was returned as-is. (2) The direct-chat fallback path did NOT inject `ACTION_TAG_DOCS` into the system prompt, so even when falling back the model had no action tag instructions. (3) `ACTION_TAG_DOCS` listed default board as `"Boards"` instead of `"Operator Board"`. Fix: (1) added `ToolMessage` presence check after `agent_executor.ainvoke` — if no tools were called, always fall through to the direct-chat path; (2) computed `_action_docs_block` in the direct-chat path which injects `ACTION_TAG_DOCS` when `needs_agent=True`; (3) fixed default board name; (4) added explicit instruction in `ACTION_TAG_DOCS` that "remind me today/later" must always emit `CREATE_TASK`. Rule: NEVER trust a LangGraph agent response as final unless `ToolMessage` objects appear in `result["messages"]`. Always fall through to direct-chat with `ACTION_TAG_DOCS` injected when the agent produced no tool calls.

- **Voice handler broke cross-channel history sync (telegram.py `handle_voice`)**: `handle_voice` used in-memory `chat_histories[chat_id]` for LLM context instead of `get_global_history`, and never called `save_global_message`. This meant all voice transcripts and Z replies from voice were completely invisible to the dashboard and to future text-based LLM context. Fix: replaced `chat_histories[chat_id]` with `await get_global_history(limit=10)`, added `save_global_message("telegram", "user", transcript)` before the LLM call and `save_global_message("telegram", "z", clean_reply, model=...)` after `parse_and_execute_actions`. Rule: EVERY Telegram input handler (text, voice, callbacks) MUST call `save_global_message` for both user input and Z's reply to maintain the unified `global_messages` timeline. Never use in-memory `chat_histories` for context — always use `get_global_history` from the DB.

- **Dashboard ChatPrompt never re-synced after successful send (ChatPrompt.ts)**: After `handleSend()` completed successfully, the assistant message was pushed to the local `this.messages` array but `loadHistory()` was never called again. Any Telegram messages that had arrived since the last history fetch were absent until the user switched tabs or focused the window. Additionally, `loadHistory()` used a hardcoded `limit=20` meaning the message list could shrink after a history refresh. Fix: (1) added a non-blocking `this.loadHistory()` call immediately after the `this.messages.push({ role: 'assistant', ... })` line so the display re-syncs with the DB after every successful send; (2) changed the fetch limit to `Math.max(30, this.messages.length + 5)` so the limit grows with the conversation. Rule: after any successful dashboard chat send, always call `loadHistory()` (non-blocking) to merge the authoritative DB state including interleaved Telegram messages.

- **Systematic hallucination in 9 of 11 action tag handlers**: Every handler in `parse_and_execute_actions` except CREATE_TASK and MOVE_CARD/MARK_DONE could silently confirm success to the user even when the backend call failed or threw an exception. Three failure-mode classes: (A) `try/except` that only called `logger.error/warning` with no `⚠` appended to `executed_cmds` (REMIND, SCHEDULE_CUSTOM, SET_NUDGE_INTERVAL); (B) return value from service layer checked for truthy/bool but no `⚠` emitted on failure path (CREATE_PROJECT, CREATE_BOARD, CREATE_LIST, CREATE_EVENT); (C) no `try/except` at all — an exception would crash the whole tag-processing loop silently (ADD_PERSON, LEARN, PROXIMITY_TRACK). Fix: wrapped every handler in `try/except Exception`, checked all return values, and appended a `⚠` failure notice to `executed_cmds` on every error path. The existing `failure_notices` surfacing logic then ensures `⚠` notices are always appended to the user-visible reply even when the LLM text already said "Done". Also fixed the `create_project` `@tool` function itself. Rule: EVERY action tag handler MUST (1) be wrapped in `try/except`, (2) check the return value of the service call, and (3) append a `⚠` string to `executed_cmds` on ANY failure path. Never use bare `logger.error` as the sole error response.

- **Mixed tab/space corruption when editing @tool functions with replace_string_in_file**: When editing `create_project` in `agent_actions.py`, the new string used tabs (agents.md tab rule) while the file's `@tool` functions use 4-space indentation — causing `TabError`. Separately, the old string consumed the `create_project` body AND ate the following `@tool\nasync def create_event(...)` decorator+signature, leaving `create_event`'s body floating as a second docstring inside `create_project`. Fix: reconstructed both functions in one replacement using 4-space indentation. Rule: for Python files always match the EXISTING indentation style. Before any replacement on a multi-function block, verify old string ends at a blank line before the next `@` decorator. Run `python -c "import ast; ast.parse(open(f).read())"` immediately after every Python edit.

- **LLM hallucinated task/card confirmation with no backend execution**: When user said "Job application sent. Remind me today to order presents for son's upcoming birthday." Z responded "Done — task added" but neither action executed. Two root causes: (1) No MOVE_CARD / MARK_DONE action tag existed anywhere — planka.py had no move_card() function, agent_actions.py had no handler, and ACTION_TAG_DOCS in llm.py never mentioned these tags, so Z had no tool to move a card to Done even if it wanted to. (2) Silent CREATE_TASK failure — when planka_create_task() returns False, executed_cmds stayed empty but the LLM's success text still reached the user unchanged. Fix: added planka.move_card() (case-insensitive card name fragment search across all boards + PATCH listId), added [ACTION: MOVE_CARD | CARD: fragment | LIST: col | BOARD: optional] and [ACTION: MARK_DONE | CARD: fragment] tag handlers in agent_actions.py (with ⚠ failure notice on False), added move_card tool to AVAILABLE_TOOLS, documented both tags in ACTION_TAG_DOCS with intent examples ("job application sent", "email sent"). Also fixed silent CREATE_TASK failure to surface ⚠ notices appended to clean_reply. Rule: any user statement that implies a task is done/sent/finished MUST trigger MARK_DONE. Any action tag that fails MUST surface a ⚠ notice to the user — never silently swallow failures.

- **CronTrigger missing explicit timezone caused 1-hour-late morning briefing**: `CronTrigger(hour=brief_hour, minute=brief_min)` was created without a `timezone=` argument. APScheduler 3.x falls back to `tzlocal.get_localzone()` for the trigger's timezone when none is provided; on the Linux VPS this is UTC. With the user in CET (UTC+1), `CronTrigger(hour=6)` fired at 06:00 UTC = 07:00 local — exactly 1 hour late. Fix: pass `timezone=pytz.timezone(user_tz_str)` explicitly to all fixed-time `CronTrigger` instances (morning_briefing, weekly_review, monthly_review, quarterly_review, yearly_review). Secondary fix: in `main.py`, call `refresh_user_settings()` BEFORE `start_scheduler()` so `_cached_timezone` is populated from the DB identity record at boot time rather than falling through to the hardcoded `"Europe/Berlin"` default. Rule: always pass an explicit `timezone=` to every `CronTrigger`; never rely on scheduler inheritance or OS locale for time-sensitive jobs.

- **`require_auth` crashed with TypeError on unauthenticated requests, breaking RAM widget**: `hmac.compare_digest(provided, token)` raised `TypeError: unsupported operand types: 'NoneType' and 'str'` whenever a request arrived without any Authorization header, query param, or cookie, leaving `provided=None`. Every unauthenticated poll of `/api/dashboard/server-info` returned HTTP 500, so the HardwareMonitor RAM bar showed 0% instead of the real value. Fix: added an explicit `if not provided: raise HTTPException(status_code=401) from None` guard before the `hmac.compare_digest` call. Rule: always guard `hmac.compare_digest` with a `not provided` check first — passing None to either argument raises TypeError, never a security-safe False.

- **Calendar widget never refreshed after agent CREATE_EVENT (ChatPrompt.ts)**: `parse_and_execute_actions` returns human-readable strings like `"Event 'X' scheduled."` as `executed_cmds`. The dashboard API passes these directly in the `actions` field of the response. `ChatPrompt.ts` dispatched `refresh-data` with `detail: { actions: data.actions }` (the raw strings). `CalendarAgenda` listens for `e.detail.actions.includes('calendar')` — a literal string match that never matched a human-readable description. Fix: added a mapping step in `ChatPrompt.ts` that converts action strings to widget refresh keys (`/event|scheduled|calendar/i` → `'calendar'`, `/task/i` → `'task'`, etc.) before dispatching `refresh-data`. Rule: always map human-readable action result strings to semantic widget keys before dispatching `refresh-data` events; never rely on natural-language substrings matching identifiers.

- **Retry button selector broken due to column-reverse DOM order (ChatPrompt.ts)**: The `finally` block of `handleSend()` attached the retry click handler with `querySelector('.message.assistant:last-child .bubble')`. With `display: flex; flex-direction: column-reverse`, rendering is done via `container.innerHTML = [...this.messages].reverse().map(...)` — the newest message (the error bubble) is placed **first** in the DOM even though it appears visually at the **bottom**. The `:last-child` selector therefore found the oldest assistant message, not the error bubble. Fix: changed selector to `:first-child`. Rule: in column-reverse message lists, the newest DOM child is always the FIRST, so always use `:first-child` to target the latest message.

- **WCAG AA color-contrast failures from un-overridden :root background-color**: The `:root {}` block in `tokens.css` had `background-color: hsla(228,45%,4%,1)` hardcoded. Light-mode overrides in `[data-theme="light"]` and `@media(prefers-color-scheme:light)` set `--bg-body` to near-white but never set `background-color` directly. axe-core walks up the DOM ancestor chain to compute effective background of inline elements; all transparent overlays composed against the hardcoded dark `html` background, producing near-identical dark foreground/background colors (e.g. `#06070f` text on `#12131b` bg). Fix: changed `:root { background-color: ... }` to use `var(--bg-body)` so it properly reflects the theme. Rule: any explicit color value in `:root` that must differ between themes MUST use a token variable, not a hardcoded `hsla()`.

- **CSS custom property cascade into Shadow DOM requires per-component flush**: Chrome defers resolving CSS custom property inheritance independently for each shadow root after attribute/media changes. The accessibility test had a `getComputedStyle` flush for `z-personality` only; other components (`user-card`, `system-benchmark`) still had stale token values when axe ran, showing 40% L accent color instead of the 22% L light-mode override. Fix: iterate `document.querySelectorAll('*')` and call `getComputedStyle` on a child of every element that has a `shadowRoot`. Rule: whenever the accessibility test changes theme (emulateMedia or data-theme attribute), flush ALL shadow roots, not just one.

- **:host-context() removed in Chrome 132**: Shadow DOM components that used `:host-context([data-theme="light"])` as a light-mode override guard found those rules entirely ignored (invalid selector → whole ruleset skipped). The correct pattern is to rely on CSS custom property token cascade from the document root (which works after the flush above), not on `:host-context`.

- **JavaScript inline styles (style.setProperty) override CSS with higher specificity than any stylesheet rule including media queries**: `UserCard.applyToRoot()` calls `document.documentElement.style.setProperty('--accent-primary-l', ...)` every time the component initialises, writing inline styles to the `<html>` element. Inline styles win over `[data-theme="light"]` and `@media(prefers-color-scheme:light)` rules. In tests, this caused the old default 38% L teal applied as an inline style to permanently override the 40% L value in `tokens.css`. Root cause chain: (1) UserCard default palette had `hsla(196,78%,38%,1)` hardcoded, (2) inline style override defeated CSS cascade, (3) Playwright accessibility test light-theme check saw wrong color. Fixes: (a) update UserCard hardcoded defaults/fallbacks from 38% to 40% L, (b) in the light-theme contrast test, call `root.style.removeProperty(...)` for each inline CSS custom property before axe runs so the CSS cascade takes effect.

- **Chrome Shadow DOM CSS custom-property cascade fails for 3-level nested var() chains**: When `--accent-color: var(--accent-primary)` and `--accent-primary: hsla(var(--accent-primary-h), var(--accent-primary-s), var(--accent-primary-l), 1)`, Chrome failed to cascade the resolved `--accent-color` value into Shadow DOM elements even when a `getComputedStyle` flush was applied. Simple flat custom properties (one-level) cascaded correctly but 3-level chains did not. Fix: define `--accent-color` as a flat literal `hsla(196, 78%, 40%, 1)` in tokens.css `:root` rather than via `var(--accent-primary)`.

- **Playwright browser contexts share localStorage with the developer's Chrome profile for localhost origins**: When Playwright runs tests against `http://localhost:4173` using the downloaded Chromium browser, `z_theme` saved to localStorage by a prior developer session persisted across test runs. The `index.html` instant-theme inline script reads this on every page load and calls `style.setProperty` with stale colors before any CSS loads. Fix: add `storageState: { cookies: [], origins: [] }` to the project `use` config in `playwright.config.ts` AND add `page.addInitScript(() => localStorage.removeItem('z_theme'))` in `test.beforeEach` before `page.goto()`. Both are needed: config clears at context creation; initScript clears mid-session re-navigations.

- **--color-danger contrast on dark card surface**: Dark mode `--color-danger: hsla(0,84%,60%,1)` gave only 4.25:1 on the dark teal `--bg-surface` (`#132423`). - **--accent-primary too light for text on near-white surfaces**: The light-mode accordion for `--accent-primary-l: 35%` gives `hsl(173,80%,35%)` which has luminance ≈ 0.266, yielding only 2.91:1 on near-white card surfaces. - **User-controlled data in log messages (log injection, CWE-117)** (CodeQL #32, #225-227, #213): Several logger calls in memory.py passed `distilled_text` and `clean_input` (both derived from user messages) directly to `%s` format arguments without stripping newlines.
