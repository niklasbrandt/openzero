# Agent Behavior & Correction Log

This file tracks policy violations, behavioral corrections, and manual overrides of AI agent actions in this repository. Agents MUST read this log at the beginning of every interaction to learn from past mistakes.

## Violations & Corrections

- **Missing database migration for new ORM columns**: Added `color_primary`, `color_secondary`, `color_tertiary` columns to the `Person` SQLAlchemy model but never ran a corresponding `ALTER TABLE` migration. The live PostgreSQL database did not have these columns, causing every query to the `people` table to crash with `UndefinedColumnError`. This broke both the UserCard widget and the Life Overview widget. Fix: ran `ALTER TABLE people ADD COLUMN IF NOT EXISTS color_primary VARCHAR; ...` directly on the VPS postgres container. Rule: whenever a new column is added to an ORM model (db.py), **always** immediately write and run a matching SQL migration (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS ...`) on the live database. Never leave the DB schema behind the ORM model.

- **TabError in Python files**: Agent used tabs for new code in `config.py` (per agents.md tab rule) but Python enforces consistency within a file. The file already used 4-space indentation, so mixing tabs caused `TabError: inconsistent use of tabs and spaces`. Rule: For Python files, match the existing indentation style (spaces) even though agents.md says tabs. Python's strict indentation rules override.

- **get_nav_markup required-arg crash broke morning briefing**: `get_nav_markup()` had its signature changed to `t: dict` (required, no default) during a refactor, but `morning.py` was still calling `get_nav_markup()` with no arguments, crashing every morning briefing with `TypeError: get_nav_markup() missing 1 required positional argument: 't'`. Fix: made `t` optional (`t: dict = None`) with an English fallback inside the function, and updated `morning.py` to pass the resolved `t` dict. Rule: when changing a function signature from optional to required (or vice versa), always search all call sites across the codebase and update them atomically in the same commit. Never leave callers behind.

- **Wrong HuggingFace model URLs**: Used `microsoft/phi-4-mini-instruct-gguf` which is gated (returns "Invalid username or password"). Correct ungated source: `unsloth/Phi-4-mini-instruct-GGUF`. Also used non-existent filename `Meta-Llama-3.1-8B-Instruct-Q4_0.gguf` in bartowski repo -- the correct filename is `Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf`. Always verify HuggingFace GGUF filenames exist before configuring.

- **Wrong Docker image registry**: Used `ghcr.io/ggerganov/llama.cpp:server` which does not exist. The llama.cpp project moved to `ghcr.io/ggml-org/llama.cpp:server`. Always verify Docker image availability before deployment.

- **Wrong binary path in container**: Assumed `llama-server` was on PATH inside `ghcr.io/ggml-org/llama.cpp:server`. The binary is at `/app/llama-server`. Always inspect container filesystem before writing entrypoints.

- **Mismatched env var names for LLM threads**: docker-compose.yml referenced `LLM_THREADS_INSTANT`, `LLM_THREADS_STANDARD`, `LLM_THREADS_DEEP` but .env defined `LLM_INSTANT_THREADS`, `LLM_STANDARD_THREADS`, `LLM_DEEP_THREADS`. The reversed naming caused all containers to silently fall back to defaults (2/2/4 threads instead of 4/4/6). Always ensure docker-compose env var references match the exact names in .env and .env.example.

- **Theme flash not eliminated by prior attempt**: Previous fix applied `no-transition` in the inline `<head>` script and added a `#page-loader` overlay, but dismissed the loader on `DOMContentLoaded` without waiting for the `initTheme()` async API call to settle. The fetch to `/api/dashboard/personality` resolved after the loader had already faded, causing CSS vars to change on visible content — the observable flash. Fix: `initTheme()` now wraps all `setProperty` calls in a `no-transition` guard with a double-`requestAnimationFrame` release; the page-loader is gated on `Promise.all([domReady, Promise.race([themeReady, 1.5s timeout])])` so content is never revealed before the final palette is committed to the DOM. Rule: any async operation that modifies CSS vars must either complete before content is visible, or use `no-transition` to suppress the repaint.

- **Heredoc terminal contamination corrupts source file**: Used `cat << 'TSEOF'` in `run_in_terminal` to write a large TypeScript file. The terminal tool injected the previous command's text (a git command) verbatim into the middle of the file at the point where the prior output buffer merged. This caused a TypeScript build error (`error TS1005: ',' expected`) on the corrupted line. Rule: never write large source files via heredoc in `run_in_terminal`. Use `create_file` or `replace_string_in_file` tools for all file writes. If the file already exists and must be fully replaced, remove it first with `rm` then use `create_file`.

- **SECRET LEAK: .env.remote committed to git**: The file `.env.remote` containing real Telegram bot token, database password, Planka admin credentials, Qdrant API key, Pi-hole password, personal email, and Tailscale IP was tracked by git and pushed to the remote repository. The file was not listed in `.gitignore`. Fix: removed from git tracking (`git rm --cached`), added to `.gitignore`. All exposed secrets should be rotated since they exist in git history. Always verify that any file containing real credentials is covered by `.gitignore` before committing.

- **Syntax error in startup greeting caused crash loop**: Added a startup greeting to `main.py` with a malformed `try/except` block (`except` on same line as `logging.info`). This caused the backend container to crash-loop repeatedly. The startup greeting was also a duplicate of one already in `telegram.py`. Rule: Never add duplicate functionality without checking if it already exists. Always verify syntax before committing code to critical startup paths.

- **Scheduler timezone double-init**: `scheduler.py` initialized `AsyncIOScheduler` at module level with `get_user_timezone()` which always returned the fallback "Europe/Berlin" because the DB cache wasn't populated yet (populated later by `refresh_user_settings()` in the startup sequence). Then `start_scheduler()` reconfigured it with the correct timezone from DB. Fix: initialize with `pytz.utc` at module level since `start_scheduler()` always reconfigures. Rule: Never rely on DB-cached values at module-load time; they aren't populated until the async startup sequence runs.

- **Streaming path silently missing birthday tags**: `chat_stream_with_context().fetch_people()` was a ~140-line duplicate of `chat_with_context().fetch_people()` but the streaming copy was missing birthday proximity tags. The duplication caused silent feature drift. Rule: When logic is duplicated, extract shared helpers to avoid divergence.

- **static/dashboard-assets mount crash on startup**: `main.py` checked `os.path.exists("static")` before mounting, but unconditionally tried to mount `static/dashboard-assets/`. When Vite inlines all assets into `index.html` (small app, no chunked JS), the `dashboard-assets/` subdirectory is never created. The `static/` dir existed (contains only `index.html`), so the outer guard passed, then `StaticFiles(directory="static/dashboard-assets")` raised `RuntimeError: Directory does not exist` and the backend crash-looped. Fix: split the guard — mount `/dashboard-assets` only `if os.path.exists("static/dashboard-assets")`, and mount `/static` separately `if os.path.exists("static")`. Rule: always guard each `StaticFiles` mount against its own specific directory, not the parent.

- **LLM container healthchecks checking wrong port**: `ghcr.io/ggml-org/llama.cpp:server` image has a built-in healthcheck that probes `localhost:8080`. openZero runs llm-instant/standard/deep on ports 8081/8082/8083 respectively. The image's built-in healthcheck was never satisfied, leaving all three containers permanently `unhealthy` despite being fully functional. Fix: override with explicit `healthcheck` blocks in docker-compose.yml pointing to the correct port for each service. Rule: Always override image-level healthchecks in docker-compose when the container port differs from the image default.

- **Stray backtick in template literal broke TypeScript build**: During the accessibility pass on `LifeOverview.ts`, a replacement introduced a stray backtick at the start of a template literal inner expression (`` `<div class="timeline-item"... ``), terminating the outer template literal prematurely. This caused 20+ TypeScript errors (TS1005, TS1128, TS1161) and broke the Docker build. Fix: remove the extra backtick. Rule: When replacing lines inside template literals (backtick strings in TypeScript/JavaScript), always verify the replacement does not introduce an unmatched backtick that would terminate the outer string context before committing.

- **Language directive ignored for briefings, greeting, and email summaries**: `chat_stream()` in `llm.py` skips `build_system_prompt()` entirely when a `system_override` is provided, so the `lang_directive` was never appended to calls made by the morning briefing, startup greeting, email poller, and any scheduled task that passes its own system prompt. Z always responded in English regardless of the configured language. Fix: when `system_override` is provided, `chat_stream()` now checks `user_profile.get("language")` and, if absent or English, calls the new `_get_user_lang()` helper which reads the identity record and caches the result in `_cached_user_lang`. The language directive is appended to the override before the LLM call. Rule: any code path that calls `chat()` or `chat_stream()` with `system_override=` bypasses the normal system-prompt builder — always verify that language (and any other user-profile-driven behaviour) is explicitly preserved in that path.

- **Missing `logger` definition caused silent startup greeting failure**: During the security hardening pass (L-A1), `print(DEBUG...)` calls in `llm.py` and `logger.warning()` calls added to `telegram.py` referenced `logger` at the module level, but neither file had `logger = logging.getLogger(__name__)` defined (only `import logging` in telegram.py; no logging import at all in llm.py). On every container start, `chat()` in llm.py hit the `logger.debug(...)` at line 310 within 31ms and raised `NameError: name 'logger' is not defined`, which was swallowed by the outer `except Exception` in `send_startup_greeting`. Z silently failed to greet after every deploy. Fix: added `import logging` + `logger = logging.getLogger(__name__)` to both files. Rule: whenever converting any `print()` to `logger.*()` in a file, immediately verify that `logger = logging.getLogger(__name__)` exists at module scope in that SAME file before committing. A global syntax check (`ast.parse`) does not catch NameErrors — also grep each modified file for `logger\.` to confirm the definition is present.

- **Mobile dashboard unreachable via open.zero (Tailscale Split DNS)**: Dashboard was reachable on the local machine but returned `ERR_NAME_NOT_RESOLVED` on mobile for `open.zero`. Root cause: Tailscale Split DNS entry for `zero` domain pointing to the Pi-hole resolver was not propagated to the mobile client, either because the entry was missing in the Tailscale admin console or the mobile Tailscale session was stale and had not picked up the DNS config. No code change required. Fix: verify the Split DNS entry exists in Tailscale admin (DNS tab → Nameservers → add the Pi-hole Tailscale IP as a restricted nameserver for domain `zero`), then force-reconnect Tailscale on mobile (toggle off/on). Rule: when `open.zero` stops resolving on any device, always check Tailscale Split DNS config first and reconnect the client before investigating application code.

- **planka.py try/except indentation mismatch caused startup crash**: During the security hardening pass converting `print(DEBUG)` calls to `logger.debug()`, the `except Exception as e:` block in `create_task()` was left at 2-tab indentation (matching the `async with` block) instead of 1-tab (matching the `try:` block). Python raised `SyntaxError: invalid syntax` on that line because `async with` is not a `try/except` statement. The backend container crashed on startup. Fix: corrected indentation to 1 tab. Rule: After any replacement that touches `try/except` blocks or any block spanning an `async with` context manager, always run `python3 -c "import ast; ast.parse(open(path).read())"` on ALL modified service files (not just the API files) before committing.

- **Shadow DOM button drift from independent per-component CSS**: Every Shadow DOM Web Component had its own hand-written button CSS with wildly different values — border-radius ranged from 4px to 10px, font-size from 0.7rem to 0.85rem, padding from `2px 6px` to `0.75rem`, and hover patterns were all inconsistent. Root cause: global `style.css button {}` rules are completely invisible inside Shadow DOM. Fix: created `src/dashboard/services/buttonStyles.ts` with a `BUTTON_STYLES` constant defining the full button design system (base, `.btn-primary/.save-btn`, `.btn-ghost/.cancel-btn`, `.delete-btn/.btn-danger`, `.btn-sm`), then imported and injected `${BUTTON_STYLES}` at the start of each component's `<style>` block. Rule: For Shadow DOM components, NEVER define button CSS independently — always import and inject the shared `BUTTON_STYLES` constant.

- **Template literal deep-indentation mismatch on multi_replace**: CalendarManager.ts has its `shadowRoot.innerHTML` template literal starting from 3-tab depth, with `<style>` at 8 tabs inside the template literal. An attempt to match `\t\t\t\t\t\t\t\t\t<style>` (9 tabs) failed because the actual file has 8 tabs. Rule: When doing string replacements in deeply-nested template literals, always verify exact tab count with `python3 -c "repr(line)"` before writing the replacement pattern.

- **Security: CORS wildcard allowed all origins**: `main.py` had `allow_origins=["*"]` in CORSMiddleware, meaning any site could make credentialed cross-origin requests to the API. Fix: changed to `allow_origins=[settings.BASE_URL]`. Rule: Never ship wildcard CORS in a backend that handles private data. Always scope the origin allowlist to the known frontend URL.

- **Security: Redis had no password**: The Redis container ran without authentication. Any container in the internal Docker network could read or inject into the task queue. Fix: added `REDIS_PASSWORD` env var, configured Redis with `--requirepass`, updated Celery broker/backend URLs to include the password via a helper that omits credentials when the password is blank (safe for local dev). Rule: Always require authentication for queue/cache services even inside an internal Docker network.

- **Security: Telegram callback handlers missing @owner_only**: `handle_unlearn_approval`, `handle_wipe_confirm`, `handle_approval`, and `handle_calendar_approval` were registered as CallbackQueryHandlers without the `@owner_only` guard. An attacker who knew the callback data format could trigger destructive actions (wipe memory, approve cloud disclosure) by sending crafted callbacks to the bot. Fix: applied `@owner_only` to all four handlers. Rule: Every handler registered with the Telegram bot (command OR callback) must be wrapped with `@owner_only`.

- **Security: Docker images pinned to :latest**: `qdrant`, `pihole`, `planka`, `whisper`, and `openedai-speech` all used `:latest`, making builds non-reproducible and vulnerable to silent breaking changes. Fix: pinned to specific stable versions (qdrant:v1.17.0, pihole:2024.07.0, planka:1.24.2, whisper:v1.9.1, openedai-speech:0.18.2). Rule: All production Docker images must use explicit version tags. Document update cadence (`docker compose pull` + test cycle) before bumping versions.

- **Qdrant and Planka version downgrade crashed storage**: When pinning from `:latest`, both `qdrant/qdrant:v1.13.0` and `ghcr.io/plankanban/planka:1.24.2` were chosen without verifying the actual running version. Both were older than what wrote the live data: qdrant `:latest` was v1.17.0, and planka `:latest` was v2.0.3 (a full major version jump — Planka v1.24.x vs v2.x). Both containers crashed on startup with migration/deserialization errors. Fix: pinned to the exact running versions (qdrant:v1.17.0, planka:2.0.3). Rule: Before pinning a service that persists state (qdrant, postgres, planka, redis), **always verify the actual running version first** (`docker run --rm <image>:<current-tag> --version` or inspect the image label). Never guess a version tag — always confirm it matches what wrote the live data. A major version downgrade (v2.x → v1.x) is especially catastrophic.

- **UFW port 53 not opened for Tailscale -- mobile DNS broke**: Pi-hole resolves `open.zero` for all Tailscale peers via Tailscale Split DNS, but the host firewall (`ufw`) had no rule allowing port 53 inbound on the `tailscale0` interface. UFW default incoming policy is `deny`, so mobile Tailscale clients could not reach the Pi-hole DNS resolver. The desktop was unaffected because it resolved via a different path. Root cause: when Pi-hole uses `network_mode: host`, DNS queries arrive at the host on port 53 -- not inside the Docker bridge -- and are subject to UFW rules. Fix: added source-restricted rules `ufw allow in on tailscale0 from 100.64.0.0/10 to any port 53 proto udp` and `proto tcp` (Tailscale CGNAT range only). Rule: whenever Pi-hole (or any host-networked DNS service) is in the stack, always verify port 53 is explicitly allowed on `tailscale0` from `100.64.0.0/10` in UFW. Always prefer the source-restricted form over a broad `to any` rule.

- **TabError in dashboard.py auth_router block crashed backend**: Added `auth_router` block to `dashboard.py` using tabs (per agents.md rule), but `dashboard.py` uses 4-space indentation throughout. Python raised `TabError: inconsistent use of tabs and spaces` on import, crashing the backend on every container start and breaking the dashboard on all devices. Fix: converted the new block to 4-space indentation to match the existing file. Rule: For Python files, always match the existing indentation style -- `dashboard.py` uses spaces, not tabs. Run `python3 -c "import ast; ast.parse(open(path).read())"` after every edit to a Python file before committing.

- **`/protocols` crashed with `TypeError: 'StructuredTool' object is not subscriptable`**: The `/protocols` handler in `dashboard.py` iterated `AVAILABLE_TOOLS` using dict-style subscript access (`t['name']`, `t['description']`), but `AVAILABLE_TOOLS` contains `StructuredTool` objects (LangChain), not dicts. Every `/protocols` request returned 500. Fix: changed to attribute access (`t.name`, `t.description`). Rule: when iterating `AVAILABLE_TOOLS`, always use attribute access — `StructuredTool` objects are not subscriptable.
- **Z ignores what user already said -- no active listening directive**: Z repeatedly asked the user questions they had already answered ("Hast du ihm schon gratuliert?" after the user said "ich habe christian gratuliert"). Three root causes: (1) the system prompt had no active listening / conversational coherence directive, so the local model freely contradicted or re-asked things the user stated; (2) `_build_history_text()` truncated Z's own responses to 200 characters, losing conversational context for follow-up messages; (3) no emoji prohibition despite agents.md rule, so the model over-decorated responses with emojis making them look unserious. Fix: added "ACTIVE LISTENING -- CRITICAL" section to `SYSTEM_PROMPT_CHAT` with explicit rules (treat user statements as facts, never re-ask answered questions, accept corrections, don't over-interpret simple statements), added "NO EMOJIS" rule, increased Z's history truncation from 200 to 500 chars. Rule: any conversational AI system prompt must include explicit active listening directives -- small local models do not infer this behavior from tone alone.

- **Pi-hole gravity.db schema corruption broke DNS for all clients**: The Pi-hole gravity database (`/etc/pihole/gravity.db`) became corrupted with a schema mismatch (`table adlist has 10 columns but 12 values were supplied`, `no such table: OLD.domain_audit`). Pi-hole FTL crashed on every DNS query while trying to check the gravity DB, causing all DNS resolution -- including `open.zero` -- to time out for every Tailscale peer. The container showed healthy and `pihole status` reported FTL listening, masking the actual failure. Fix: deleted the corrupt `gravity.db` and `gravity.db_temp` inside the container, ran `pihole -g` to rebuild from scratch. The custom DNS records (`custom.list`) are stored separately and were unaffected. Rule: if DNS suddenly stops resolving for all clients but Pi-hole reports healthy, immediately check `FTL.log` for `gravityDB` errors. Fix is `rm /etc/pihole/gravity.db && pihole -g`.
